# ZeninDB Architectural Specification

ZeninDB is an embedded, local-first, eventually consistent database for collaborative applications. Every device holds a full replica, works offline indefinitely, and converges when it reconnects. No central server required.

---

## 1. Core Principles

### P1 — Per-cell freshness via HLC

Every addressable value is a `Cell`. Each Cell has a Hybrid Logical Clock. A cell's HLC reflects only the last direct write to that exact cell. HLCs do not cascade upward when a child changes. Used for LWW comparisons, ordering, and tombstone resolution.

### P2 — Per-cell sync policy

`Cell.sync` is `Option<bool>`: `None` (inherit from parent), `Some(true)` (synced), `Some(false)` (local-only). A child can override its parent. `Delta.sync` is always a plain `bool`, resolved at write time from the `Option<bool>` chain. `SetSync` toggles are local-only — they never replicate.

### P3 — Self-healing recursive structure

The database is a recursive tree of typed Cells. Operations carry type information in their paths, so any peer can apply an operation even when intermediate structure hasn't been observed locally. Missing parents are materialized as dummy cells and reconciled later.

### P4 — Compacted state is the replication truth

The WAL supports aggressive compaction. It is used for crash recovery, reactive events, and recent live sync — not for long-term history. Convergence uses state-based anti-entropy: peers compare state-hash indexes over compacted state and exchange differing chunks.

### P5 — Transport agnostic

The database handles serialization, signing, conflict resolution, and merge. The embedder provides transport (TCP, QUIC, libp2p, WebSocket, files, relay server). No peer discovery or connection management in the database.

## 2. Data Model

### 2.1 Cell

```rust
struct Cell {
    value: Value,
    hlc:   Hlc,
    sync:  Option<bool>,
}
```

Every addressable value is a Cell. Cells form a recursive tree. A cell's HLC is the timestamp of the last operation that directly targeted it. Dummy cells (created during apply walk when parents don't exist) have `Hlc::ZERO`.

### 2.2 Hybrid Logical Clock

10 bytes, big-endian:

| Bytes | Field | Bits |
|-------|-------|------|
| 0–5 | physical_ms | 48 |
| 6–7 | logical | 16 |
| 8–9 | node_id | 16 |

Big-endian so byte comparison equals causal ordering. `Hlc::ZERO` is the all-zero sentinel. Ordering: physical desc → logical desc → node_id desc.

### 2.3 Types

Types implement a cascade of traits:

| Trait | Purpose | Implemented by |
|-------|---------|---------------|
| `TypedValue` | `encode`/`decode` | AtomValue, RecordValue |
| `TypedOp` | `encode`/`decode` | AtomOp, RecordOp |
| `TypedSegment` | `encode`/`decode` | RecordSegment |
| `Type` | `empty()`, `apply_op()`, `merge()` | AtomType, RecordType |
| `ContainerType` | `descend_or_create()` | RecordType |

Built-in types:

- **Atom** — Scalars: Null, Bool, Int, UInt, Float, String, Bytes, Timestamp, Uuid, Ulid. `AtomOp::Set(v)` with LWW.
- **Record** — Named-field container: `IndexMap<String, Cell>` + tombstones (`IndexMap<String, Hlc>`). Operations: `SetField`, `Replace`, `RemoveField`. Tombstones prevent deleted fields from being resurrected by old peers.

### 2.4 apply_op and merge signatures

```rust
fn apply_op(value: &mut Self::Value, op: &Self::Op, local_hlc: Hlc, op_hlc: Hlc)
    -> Result<bool, Self::Error>;

fn merge(local: &mut Self::Value, local_hlc: Hlc, remote: &Self::Value, remote_hlc: Hlc)
    -> Result<bool, Self::Error>;
```

Both mutate in place. Return `Ok(true)` if state was modified, `Ok(false)` if rejected (e.g. LWW loss). This signal lets the storage engine know what to persist.

### 2.5 Value, Op, Segment enums

Generated by the `register_types!` macro in `zendb-types`. One variant per registered type. `Op` includes `SetSync { sync: Option<bool> }` which is intercepted by the engine before the apply walk.

### 2.6 Path

```rust
struct PathStep {
    container_tag: TypeTag,  // expected type at this depth
    segment:       Segment,   // how to descend
}
struct Path { steps: Vec<PathStep> }
```

An empty path refers to the row root. Each step carries `container_tag` so the apply walk knows what type of dummy to create if the container doesn't exist — this is what enables self-healing.

### 2.7 Delta

```rust
struct Delta {
    table_id:    TableId,
    primary_key: PrimaryKey,
    path:        Path,
    op:          Op,
    hlc:         Hlc,
    sync:        bool,       // always a plain bool, resolved at creation
    signature:   Signature,  // opaque bytes, verified by replication crate
}
```

### 2.8 Encoding

Every type has `encode(&self, out: &mut Vec<u8>)` and `decode(bytes: &[u8]) -> (Self, usize)`. Wire format uses big-endian integers, LEB128-like varints, length-prefixed strings, and type-tag-prefixed Value/Op/Segment payloads.

---

## 3. Apply Walk

`Cell::apply_delta(&mut self, delta: &Delta) -> bool` descends recursively through the path. At each step it ensures the container has the expected type (creating a dummy if absent and HLC allows). At the leaf, dispatches to `Type::apply_op`. Returns `true` only if state was modified. HLC is bumped only when the op's HLC beats the cell's current HLC.

`SetSync` bypasses the apply walk entirely — the engine resolves the path to a Cell and sets `sync` and `hlc` directly.

---

## 4. Merge

`Cell::merge(&mut self, remote: &Cell)` combines two cells at the same path. Type mismatch → LWW on cell HLC. Type match → dispatches to `Type::merge` which mutates local in place. Sync flags merge with `Some(true)` dominating (once synced, always synced).

---

## 5. Sync Model

### 5.1 Cell sync inheritance

```
To resolve Delta.sync for a write:
  Walk from target cell upward (leaf → root)
  Return the first Some(value) found
  If none found, use database default
```

### 5.2 Isolation boundary

When effective sync resolves to `false`, the subtree is in local-only mode:
- Local writes get `Delta.sync = false` — never broadcast
- Incoming writes targeting this subtree are dropped
- Subtree excluded from state-hash index and anti-entropy

### 5.3 SetSync

`SetSync` is always local-only (`Delta.sync = false`). Sync policy is per-device. When sync is turned ON, the subtree re-enters the state-hash index; the next anti-entropy round detects divergence and `merge` converges it. When turned OFF, the subtree is removed from replicated state; peers keep syncing their version.

### 5.4 Two views

```
Local view      = synced state ∪ local-only state
Replicated view = synced state only
```

State-hash indexes, anti-entropy, and cold sync only operate on the replicated view.

---

## 6. Replication

### 6.1 Live delta sync

When a local write commits with `Delta.sync = true`:
1. Delta appended to WAL
2. Applied locally via apply walk
3. Enqueued for broadcast to reachable peers

Best-effort, low-latency. Can be missed.

### 6.2 State-based anti-entropy

Authoritative repair. Peers compare state-hash indexes over compacted synced state, exchange `StateChunk`s for differing subtrees, and merge. Does not require historical WAL entries.

### 6.3 Cold sync

New device starts empty. Streams compacted synced state from an existing peer, writes to local storage, builds state-hash indexes, transitions to live sync.

### 6.4 Transport trait

```rust
trait Transport {
    fn send_to_peer(peer_id: PeerId, bytes: Bytes) -> Result<()>;
    fn on_receive(callback: impl Fn(PeerId, Bytes));
    fn list_peers() -> Vec<PeerId>;
}
```

### 6.5 Signed deltas

Every replicated delta is signed (Ed25519). Signature covers the full payload. Verification is in the replication crate. Types crate stores signatures as opaque bytes.

---

## 7. Delete Semantics

Deletes use tombstones with HLCs. A record field is visible only if `cell.hlc > tombstone_hlc`. Tombstones are part of compacted state and included in state hashes.

---

## 8. Storage

- **WAL** — append-only delta log for crash recovery, reactive events, and live sync
- **RAM buffer** — groups recent deltas by `(TableId, PrimaryKey, Path)` before compaction
- **Compaction** — applies buffered deltas into compacted Cells, writes to physical storage, updates state-hash indexes, truncates WAL
- **Physical storage** — sorted or hash-bucket layouts, cell-level interface
- **State-hash index** — Merkle-style hashes over compacted synced state, scoped by table/row/cell

---

## 9. Reactive Events

Streaming cursors over tables, rows, paths, or query shapes. The event system tails recent deltas from the WAL and pushes matching deltas to subscribers. Local and remote deltas enter the same WAL path — subscribers observe both uniformly.

---

## 10. Type Registration

Adding a type:
1. Create one module with value/op/segment/error types
2. Implement `TypedValue`, `TypedOp`, `TypedSegment` (if container)
3. Implement `Type` (and `ContainerType` if container) on a unit struct
4. Add one line to `register_types!` in `lib.rs`

The macro generates all enum variants, dispatch functions, and the `TypeError` enum.
